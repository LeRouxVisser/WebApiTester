Getting Started

Introduction

Welcome to the 3rd party Mock system! The Solution offers creation of projects that make use of RESTful interfaces to help you mock current integrated systems of the OpenAPI system. These interfaces enable engineers and developers to continue with their work when these systems are not available or existing yet. Examples of systems that can be mocked:
  • G2
  • USSD Push
  • Any integration making use of APIs

Adding a project

Before starting to configure the RESTful APIs with its given endpoint (context), the first step is to create a project that will contain all the config. An example of this will be to create a USSD push project that will contain all the markets interfaces and endpoints. From the project screen you can access the Specs, Defects and Results pages. These pages are all bound to the project that was created.

Adding specs to a project

Access the spec page by clicking on the spec button under your created project. The spec page consists of a form with multiple inputs namely:
  • Endpoint → Specify the endpoint that will be used to send mock requests to. This endpoint will get additional info added to it namely the /user_id/project_id/.
  • Async button → Click on the async button if you want to mock an async request. You will need to specify some additional information in the form after activating the async functionality. The result URL, time before results is sent to URL and lastly the async result body in the json spec section.
  • Json spec → consists of a json nested array that contains a list with combination of requests, responses, response http status codes and lastly an async results if the async functionality is activated, otherwise it will only take 3 entries. Use "?" to add dynamic variable mapping from the request that came in on the endpoint right through the spec.
The response sent to the endpoint will be linked to the mapped requests. The matching response, response http status code and async result response (if async functionality is active) will be used to respond to the request.

Checking results on endpoints

Access the Results page by clicking on the Results button under your created project. The Results page consists of the following information of the most resent request sent to a given endpoint:
  • A status banner
    • Green if the request matched any of the mapped requests.
    • Red if the request does not match any of the mapped requests.
  • The endpoint that was created on the spec page.
  • The request that came in last and the mapped request that it matched with or all the mapped request it did not match with.

Add defect functionality on endpoint

Access the Defects page by clicking on the Defects button under your created project. The Defects page consists of the following information and inputs:
  • The endpoint that was created on the spec page.
  • Connection down function, when activated the endpoint will be switched off and requests wonâ€™t be able to get through. This will mock network connection issues.
  • Intermittent connection issues, when activated you need to specify the percentage of times the system should be down. The endpoint will be switched off given the % passed and will sometimes be active and sometimes be inactive. This will mock intermittent network connection issues.
